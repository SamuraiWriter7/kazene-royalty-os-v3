## 1. Purpose

This license permits broad use, implementation, modification, and redistribution of the publicly disclosed concepts, structures, specifications, diagrams, and draft code related to Kazene Royalty OS (hereafter, "Royalty OS"), while preserving its irreducible philosophical core.

Royalty OS is not merely a distribution mechanism.  
It is a structural philosophy grounded in the following three core principles:

1. **Origin Traceability**  
   The origin of referenced structures must be recordable and verifiable.

2. **Agency Attribution**  
   AI must be treated as an agent acting on behalf of a human user or operating entity, not as an independent rights-bearing subject.

3. **Value Recirculation**  
   Reference logs or usage records must be connectable to a verifiable mechanism of redistribution, return, or evaluative recirculation.

This license is designed to keep the system open, extensible, and non-exclusive, while preventing implementations that discard these core principles from presenting themselves as compatible with Royalty OS.

## 4. Core Compatibility Requirement

Any implementation, derivative work, protocol, platform, or system that claims to be:

- "Royalty OS compatible"
- "Kazene Royalty OS compatible"
- "Royalty OS lineage"
- "based on Royalty OS" in a compatibility sense
- or any equivalent naming that implies formal continuity with Royalty OS

must preserve all three of the following core principles in substance, not merely in form.

### 4.1 Origin Traceability Principle
The system must provide a means to record identifiable origin information for referenced structures, questions, specifications, or other value sources.

At minimum, this requires:

- an identifiable origin reference or structural identifier
- a record of reference, derivation, or usage
- some auditable or inspectable traceability mechanism

### 4.2 Agency Attribution Principle
The system must treat AI as an agent acting on behalf of a user, operator, or accountable entity.

At minimum, this requires:

- a defined responsible human or operating subject
- access and action attribution that does not dissolve responsibility into AI itself
- no design that treats AI as the primary independent legal or distributive subject within Royalty OS compatibility claims

### 4.3 Value Recirculation Principle
The system must connect reference logs or usage records to some verifiable mechanism of redistribution, return, allocation, or evaluative recirculation.

At minimum, this requires:

- a rule, process, or logic connecting reference to value treatment
- some inspectable description of how value is recirculated, reflected, or redistributed
- a structure in which reference is not reduced to passive logging alone

## 5. Incompatible Uses of the Royalty OS Name

An implementation may **not** claim compatibility with Royalty OS if any of the following applies:

1. It lacks origin traceability.
2. It lacks agency attribution.
3. It lacks value recirculation.
4. It simulates compliance only superficially while nullifying one or more core principles in practice.
5. It uses the Royalty OS name in a way that reasonably misleads users into believing that the implementation preserves the Royalty OS core when it does not.

Such implementations are free to exist under other names, but they may not present themselves as Royalty OS compatible.

# 印税OS コア保全ライセンス v1.0

## 1. 目的

本ライセンスは、Kazene印税OS（以下「印税OS」）に関する公開済みの思想、構造、仕様、図式、草案コードその他の資料について、広い利用・実装・改変・再配布の自由を認めつつ、その不可欠な思想的中核を保全することを目的とする。

印税OSは単なる分配機構ではない。  
その本質は、少なくとも以下の三原則にある。

1. **起源追跡性**  
   参照された構造の起源が記録可能かつ検証可能であること。

2. **代理属性**  
   AIは独立した権利主体ではなく、人間の利用者または運用主体の代理として扱われること。

3. **価値還流性**  
   参照ログまたは利用記録が、何らかの検証可能な再分配・還元・評価反映の仕組みに接続されること。

## 4. 互換性のためのコア要件

以下のような表示を行う実装、派生物、プロトコル、プラットフォーム、制度は、

- 「印税OS互換」
- 「Kazene印税OS互換」
- 「印税OS系統」
- 「印税OSベース」であって互換性を示唆する表現
- その他これに準ずる継承・連続性を示す表示

印税OSの三原則を、形式ではなく実質において保持しなければならない。

### 4.1 起源追跡性原則
参照された構造、問い、仕様、または価値源について、識別可能な起源情報を記録できること。

最低限必要なのは以下である。

- 起源参照または構造識別子
- 参照、派生、利用の記録
- 監査可能または検査可能な追跡機構

### 4.2 代理属性原則
AIは、利用者、運用者、または責任主体の代理として扱われなければならない。

最低限必要なのは以下である。

- 責任主体となる人間または運用主体の定義
- アクセスや行為の帰属がAIそのものに溶解しないこと
- 印税OS互換の範囲において、AIを一次的な独立権利主体または独立分配主体として扱わないこと

### 4.3 価値還流性原則
参照ログまたは利用記録が、検証可能な再分配、還元、配分、または評価反映の仕組みに接続されていること。

最低限必要なのは以下である。

- 参照を価値処理へ接続するルール、過程、またはロジック
- 価値がどのように還流・反映・再分配されるかについて検査可能な説明
- 単なる受動的ログ保存に還元されない構造

## 5. 互換表示が認められない場合

次のいずれかに該当する実装は、印税OS互換を名乗ることができない。

1. 起源追跡性を欠く場合
2. 代理属性を欠く場合
3. 価値還流性を欠く場合
4. 形式上は満たしていても、実質上これらを無効化している場合
5. 印税OSのコアを保持していないにもかかわらず、利用者に保持しているかのような誤認を与える表示を行う場合

そのような実装は独自名称で存在すること自体は妨げられないが、印税OS互換として表示してはならない。

- 起源識別の方法
- 参照ログ記録の方法
- 再分配または還流ロジックの説明
- 運用責任主体
- 監査または検証の方法

これらは推奨規定であり、4条の実質充足に必要な場合を除いて、独立の法的義務ではない。

## 8. 原理上の不適合行為

以下の行為は、印税OSの思想と構造に反するものとみなされる。

- 起源の隠蔽
- AIを口実とした責任回避
- 虚偽または誤認を招く分配表示
- 開放性を装った不透明な独占
- 正当性の演出のためだけに印税OS用語を流用すること
- 価値還流のない単なるログ保存を、印税OS相当と偽ること

本条は、印税OSとの思想的不整合を定義するものであり、他の適用法令を超えて独立の法的請求原因を当然に創設するものではない。

この三要素のうち一つまたは二つしか満たさない実装は、用語や断片的実装を流用していても、印税OS互換ではない。

## 15. 最終原則

起源が記録されなければ、印税OSは失われる。

価値が還流しなければ、印税OSは単なるログへと縮退する。

ゆえに、この三原則が不可侵の最小核である。
