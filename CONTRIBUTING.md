# Contributing to 印税OS v3.0 Core Specification

印税OS v3.0 への関心、ありがとうございます。  
このリポジトリは、AIによる著作物アクセスを  
**許諾・ログ・分配** の正規循環へ変換するための  
権利処理OS仕様を、公開・改善・発展させていくための場です。

この仕様は、単なる思いつきの断片ではなく、  
**思想・制度整合・機械可読仕様・実装可能性** をつなぐための  
オープンな骨格として設計されています。

本ドキュメントでは、参加のしかた、提案の出し方、  
改善の方向性、そして守ってほしい原則を示します。

---

## このリポジトリの目的

このリポジトリの目的は、次の4つです。

1. **公開仕様を安定的に維持すること**
2. **改善提案を受け入れ、構造を洗練させること**
3. **将来の実装・実証・制度議論に耐える骨格を育てること**
4. **思想・仕様・実装の連鎖を可視化すること**

印税OS v3.0 は、完成済みの商用プロダクトではありません。  
しかし同時に、単なる抽象理念でもありません。

ここは、  
**「現実に接続できる構造」** を共に磨く場所です。

---

## まず理解してほしいこと

印税OS v3.0 は、次の立場を取ります。

- AI利用を全面否定するものではありません
- 権利者保護だけを叫んで閉じる構想でもありません
- 投機的トークン経済を作ることが目的でもありません
- 主役はコインではなく、権利処理OSです
- Qコインは、非換金・非譲渡・内部会計単位です
- AIは独立読者ではなく、利用者または契約主体の代理として扱います
- 透明性・権利者保護・対価還元の両立を重視します

この前提から大きく外れる提案は、  
たとえ技術的に面白くても、v3.0 の範囲では採用されない可能性があります。

## 歓迎する貢献

次のような貢献を歓迎します。

### 1. 仕様改善
- YAML構造の改善
- フィールド定義の整理
- 命名の統一
- レイヤー間の整合性向上

### 2. ドキュメント改善

- 説明文の推敲
- 誤字脱字修正
- 用語集の追加

### 3. Schema / Validation
- JSON Schema の作成・改善
- バリデーションルールの追加
- フィールド制約の設計
- サンプルデータの妥当性確認

### 4. サンプル / シミュレーション
- ダミーログ生成例
- 分配レポート例
- サンプル作品データ
- Qコイン計算例
- 簡易シミュレーター

### 5. 実装補助
- API設計案
- データモデル案
- ダッシュボード案
- 監査レポート案
- UI/UX提案

### 6. 法務・制度観点の補足
- 規約上の注意点
- 著作権・契約実務上の観点
- 金融規制上の危険線の指摘
- 国際的制度との整合メモ
- 出版業界・プラットフォーム運用上の示唆

### 7. 翻訳・展開
- 英訳
- 要約版
- 発表用資料への展開
- 共同研究向け説明文の整備

## 歓迎しない方向

次のような提案は、少なくとも v3.0 では歓迎されません。

- Qコインの外部売買
- 外部ウォレット対応
- ユーザー間送金
- 暗号資産化
- ステーブルコイン化
- 投機設計
- 無許諾再学習の正当化
- ログなし利用の容認
- AI単独の無制限クロール前提設計
- 権限定義を曖昧にする変更
- 出典や系譜を消す改変

要するに、  
**「権利処理OS」ではなく「金融商品」へ寄せる方向**、  
または  
**「透明性」ではなく「抜け道」へ寄せる方向**  
は、v3.0 の思想と合いません。

## 参加方法

### 小さな修正
誤字脱字、軽微な表現改善、コメント修正などは、  
Issue なしで直接 Pull Request でも構いません。

### 中くらいの提案
フィールド追加、用語変更、説明構造の調整などは、  
先に Issue を立ててから PR を出してください。

### 大きな変更
レイヤー構造の変更、Qコイン定義の変更、  
分配原理の変更、主要方針の変更などは、  
必ず先に Discussion または Issue で提案してください。

いきなり大改造PRを投げると、構造が散りやすくなります。  
刀は鋭いほうがいいですが、いきなり天守閣まで斬らなくて大丈夫です。

---

## Issue を立てるときの目安

Issue は、次のような形式で立てると助かります。

### 推奨フォーマット
- **タイトル**  
  何を改善したいかを短く書く

- **背景**  
  なぜこの変更が必要か

- **現状の問題**  
  どこが曖昧か、どこが不足しているか

- **提案内容**  
  何をどう変えたいか

- **影響範囲**  
  README / YAML / Schema / examples / docs など

- **互換性**  
  既存仕様との整合性に問題があるか

- **補足**  
  参考資料、ユースケース、理由など

## Pull Request の基本方針

Pull Request では、次の点を意識してください。

### 1. 一度に変えすぎない
なるべく、変更の目的が分かる単位で分けてください。

### 2. 理由を書く
「なぜ変えるのか」を必ず書いてください。  
正しさだけでなく、意図の共有が大事です。

### 3. 影響範囲を明示する
どのファイルに影響するか、  
仕様のどの部分に波及するかを書いてください。

### 4. 用語を揃える
`permission`, `trace`, `allocation`, `proxy`, `QCoin` など、  
既存のキーワードと衝突しないようにしてください。

### 5. 禁止線を越えない
v3.0 の安全線・危険線に反する変更は、採用されにくくなります。

---

## ブランチ名の例

ブランチ名は、次のような形式を推奨します。

- `docs/fix-readme-phrasing`
- `spec/add-permission-flags`
- `schema/update-required-fields`
- `examples/add-sample-trace-log`
- `docs/translate-readme-en`
- `proposal/refine-allocation-coefficients`

短く、何をしているか分かる名前が望ましいです。

---

## コミットメッセージの例

コミットメッセージも、内容が分かる形を推奨します。

- `docs: refine README overview`
- `spec: add reference_only permission flag`
- `schema: require permission_snapshot field`
- `examples: add monthly allocation sample`
- `docs: add compliance notes for internal points`

## 仕様変更の優先順位

提案が競合した場合、基本的には次の優先順位で判断します。

### 優先されやすいもの
1. 構造を明確にする変更
2. 制度整合性を高める変更
3. 実装しやすさを高める変更
4. 読みやすさ・共有しやすさを高める変更
5. サンプルや検証可能性を高める変更

### 優先されにくいもの
1. 思いつきだけで追加された複雑化
2. 投機性・金融商品性を高める変更
3. 透明性を下げる変更
4. 権限・ログ・分配の骨格を壊す変更
5. v3.0 の範囲を越える飛躍的拡張

---

## v3.0 の守備範囲

この仕様は、次の範囲を主に扱います。

- 許諾モデル
- ログモデル
- 分配モデル
- 内部会計単位 Qコイン
- 代理エージェント構造
- 最小循環の設計
- ドキュメント化された参照骨格

一方で、次のものは v3.0 では主対象ではありません。

- 完全な商用規約
- 法律判断の最終結論
- 実際の決済事業設計
- 外部金融接続
- ブロックチェーン本実装
- グローバル統一規格の最終版
- 自動執行型スマートコントラクト経済圏

## 提案を書くときの視点

よい提案は、たいてい次のどれかを強くします。

- 仕様が明確になる
- 危険線が分かりやすくなる
- 実装しやすくなる
- 分配が説明しやすくなる
- 権利者にとって安心になる
- 利用者にとって分かりやすくなる
- 行政・法務・出版社にも共有しやすくなる

逆に、  
「面白いが説明しにくい」  
「派手だが制度に乗りにくい」  
「技術的には可能だが骨格がぶれる」  
ものは、v3.0 では慎重に扱います。

---

## 出典・系譜の扱い

このリポジトリでは、  
**構造の出典・系譜・文脈** を重視します。

改善や拡張を行う場合でも、  
なるべく「何を土台に何を変えたか」が分かる形を残してください。

思想の流れを消さず、  
構造の変化を見えるようにすることは、  
このリポジトリの重要な方針のひとつです。

必要に応じて、次のようなファイルの併用を推奨します。

- `CITATION.cff`
- `NOTICE`
- `CHANGELOG.md`

---

## 議論のスタンス

このリポジトリでは、  
対立のための議論ではなく、  
**構造をよくするための議論** を歓迎します。

そのため、次の姿勢を大切にしてください。

- 相手を論破するより、構造を明確にする
- 否定よりも、代案を示す
- 印象よりも、具体性を重視する
- 抽象だけでなく、実装や運用への接続を意識する
- 思想と制度の両方を尊重する

---

## 法務・制度に関する注意

このリポジトリは、法的助言や金融助言を提供するものではありません。  
法務・知財・資金決済・規制対応については、実装や事業化の段階で  
専門家の確認が必要です。

ただし、v3.0 は最初から次を重視しています。

- 規制の重い領域を避けること
- 透明性を確保すること
- 権利者保護を損なわないこと
- 実装可能性を優先すること

この方針を崩す提案は、慎重に扱われます。
